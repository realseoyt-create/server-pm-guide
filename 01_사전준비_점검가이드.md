## 1. HP-UX B.11.31 점검

> Active 서버와 Standby 서버 **각각** 동일하게 수행한다.

### 1-1. OS 기본 정보 확인

```bash
# OS 버전 확인
uname -a

# 예상 출력 예시
# HP-UX hostname B.11.31 U ia64 XXXXXX unlimited-user license

# 호스트명 확인
hostname

# 현재 일시 확인 (Active/Standby 시간 동기화 여부 확인용)
date
```

### 1-2. 디스크 사용률 확인

```bash
# 디스크 사용률 확인 (HP-UX는 bdf 사용, df -h와 동일)
bdf

# 출력 예시
# Filesystem          kbytes    used   avail %used Mounted on
# /dev/vg00/lvol3   10240000 4200000 5800000   42% /
# /dev/vg00/lvol8   20480000 8500000 11200000  43% /var
```

> ⚠️ **주의**: 사용률이 **80% 이상**인 파티션이 있으면 작업 전 반드시 정리.
> 특히 `/var`, `/tmp`, `/oracle` (또는 ORACLE 데이터 경로) 확인 필수.

```bash
# 큰 파일 찾기 (예: /var 에서 100MB 이상)
find /var -size +100000k -ls 2>/dev/null | sort -k7 -rn | head -20

# 오래된 로그 파일 확인
find /var/adm -name "*.log" -mtime +30 -ls 2>/dev/null
```

### 1-3. 메모리 및 CPU 확인

```bash
# 메모리 사용률 확인
vmstat 1 5

# CPU 사용률 확인 (5초 간격 3회)
sar -u 5 3

# 현재 실행 중인 프로세스 목록 전체 저장 (작업 전 스냅샷)
ps -ef > /tmp/pre_work_ps_$(date +%Y%m%d).txt
echo "프로세스 스냅샷 저장: /tmp/pre_work_ps_$(date +%Y%m%d).txt"
```

### 1-4. 네트워크 확인

```bash
# 네트워크 인터페이스 상태
netstat -in

# 현재 열려 있는 포트 및 연결 상태
netstat -an | grep LISTEN
netstat -an | grep ESTABLISHED | wc -l
# → 접속 세션 수 확인 (작업 전 0이 되어야 함)

# 라우팅 테이블
netstat -rn
```

### 1-5. ServiceGuard (MC/SG) 클러스터 상태 확인

> ⚠️ **중요**: HP-UX 2대가 MC/ServiceGuard로 묶여 있는 경우 반드시 확인.
> 클러스터 상태 확인 없이 서버를 내리면 **자동 Failover가 발생**하여 테스트 시나리오가 꼬일 수 있음.

```bash
# 클러스터 전체 상태 확인
cmviewcl -v

# 패키지(서비스) 상태 확인
cmviewcl -p

# 출력 예시
# CLUSTER    STATUS
# mycluster  up
#
# NODE       STATUS    STATE
# hpux01     up        running   ← Active
# hpux02     up        running   ← Standby

# 현재 어떤 패키지가 어느 노드에서 실행 중인지 확인
cmviewcl -p -v
```

> 💡 **TIP**: Failover 테스트 전에는 자동 Failover를 **수동(manual)**으로 변경해야  
> 의도치 않은 자동 전환을 막을 수 있다. (테스트 완료 후 원복 필수)

```bash
# 자동 Failover를 수동으로 변경 (작업 전)
# 반드시 관리자 확인 후 수행
cmmodpkg -d <패키지명>   # 패키지 비활성화
# 또는
cmhaltpkg <패키지명>     # 패키지 중지
```

### 1-6. 시스템 오류 로그 확인

```bash
# 시스템 에러 로그 확인
/usr/sbin/dmesg | tail -50

# 하드웨어 오류 여부 (HP-UX 전용)
/usr/sbin/cstm
# → 인터렉티브 도구. 단순 확인이면 아래 명령 사용

# 최근 부팅 이력
who -b

# 패치 레벨 확인
swlist -l product | grep PH
```

## 2. AIX 5.3 점검

### 2-1. OS 기본 정보 확인

```bash
# OS 버전 확인
uname -a
oslevel -s   # 서비스팩 레벨까지 확인

# 예상 출력
# 5300-12-XX-XXXX  (AIX 5.3 TL12 등)
```

### 2-2. 디스크 및 볼륨 그룹 확인

```bash
# 디스크 사용률 확인
df -g   # GB 단위로 표시

# 볼륨 그룹 상태 확인
lsvg
lsvg -o   # 활성 볼륨 그룹만

# 각 볼륨 그룹 상세 확인
lsvg rootvg
lsvg datavg   # (볼륨 그룹명은 환경에 따라 다름)
```

### 2-3. 오류 로그 확인 (AIX 핵심)

> 🔴 **놓치기 쉬운 포인트**: AIX는 `errpt` 명령으로 하드웨어 오류를 확인해야 한다.
> 이걸 안 보면 디스크 장애 직전 상태인데 모르고 작업하는 경우가 있다.

```bash
# 전체 오류 리포트 확인
errpt | head -30

# 상세 오류 내용 확인
errpt -a | head -100

# HARDWARE 오류만 필터링
errpt | grep -i "PERM"   # PERMANENT 오류 (심각)

# 최근 1주일 오류
errpt -s $(date -v-7d +%m%d%H%M%y 2>/dev/null || date --date='7 days ago' +%m%d%H%M%y)
```

### 2-4. 백업 서버 동기화 상태 확인

```bash
# 백업 파일 최신 여부 확인
ls -lrt /backup/   # 또는 실제 백업 경로

# 마지막 백업 일시 확인
find /backup -name "*.tar" -o -name "*.gz" | xargs ls -lt 2>/dev/null | head -10

# 백업 로그 확인 (경로는 환경마다 다름)
tail -50 /var/log/backup.log   # 또는 실제 백업 로그 경로
```

> ⚠️ **주의**: 백업 최종 성공 시점이 **24시간 이상** 지났다면 작업 전 수동 백업 수행.

## 3. Windows Web Server 점검

> ※ Windows 2대 모두 동일하게 수행.

### 3-1. IIS 서비스 상태 확인

**방법 1 - GUI**: 서버 관리자 → 도구 → IIS 관리자 → 사이트 상태 확인

**방법 2 - CMD (명령 프롬프트, 관리자 권한)**:
```cmd
:: IIS 서비스 상태 확인
sc query W3SVC

:: 실행 중인 사이트 목록
%systemroot%\system32\inetsrv\appcmd.exe list sites

:: 애플리케이션 풀 상태
%systemroot%\system32\inetsrv\appcmd.exe list apppool /processModel.userName:applicationPoolIdentity
```

### 3-2. 디스크 확인

```cmd
:: 디스크 사용률 확인
wmic logicaldisk get caption,freespace,size

:: C 드라이브 사용률이 90% 이상이면 로그 정리
:: IIS 로그 위치: C:\inetpub\logs\LogFiles\
dir C:\inetpub\logs\LogFiles\ /s
```

> 🔴 **놓치기 쉬운 포인트**: IIS 로그가 쌓여서 C 드라이브가 꽉 찬 경우가 많다.
> 작업 전에 오래된 로그 정리 또는 로그 경로를 D 드라이브로 변경할 것.

### 3-3. 이벤트 로그 확인

```cmd
:: 시스템 오류 이벤트 최근 20개
wevtutil qe System /c:20 /rd:true /f:text | findstr /i "error critical"

:: 애플리케이션 오류 확인
wevtutil qe Application /c:20 /rd:true /f:text | findstr /i "error"
```

## 4. ORACLE DB 점검

> HP-UX Active 서버에서 `oracle` 계정으로 접속하여 수행.

### 4-1. ORACLE 프로세스 및 리스너 확인

```bash
# oracle 계정으로 전환
su - oracle

# ORACLE 프로세스 확인
ps -ef | grep pmon   # pmon 프로세스가 있으면 DB가 올라온 것

# 리스너 상태 확인
lsnrctl status

# 예상 출력 (정상 시)
# LSNRCTL for HP-UX: ...
# Connecting to (ADDRESS=(PROTOCOL=tcp)(HOST=...)(PORT=1521))
# STATUS of the LISTENER:
# Alias                     LISTENER
# Version                   ...
# Start Date                ...
# Uptime                    X days X hr. X min. X sec
# ...
# Services Summary...
# Service "ORCL" has 1 instance(s).   ← 서비스 등록 확인
```

> 🔴 **놓치기 쉬운 포인트**: DB 기동 후 `lsnrctl status`에서 서비스가 등록되어 있는지 반드시 확인.
> DB는 올라왔는데 리스너에 서비스가 등록 안 된 경우 → 앱에서 ORA-12541 오류 발생.

### 4-2. DB 인스턴스 상태 확인

```bash
# sqlplus 접속
sqlplus / as sysdba

-- DB 상태 확인
SELECT STATUS FROM V$INSTANCE;
-- 결과: OPEN 이어야 정상

-- DB 모드 확인
SELECT NAME, OPEN_MODE FROM V$DATABASE;
-- 결과: OPEN_MODE = READ WRITE 이어야 정상

-- 마운트된 데이터파일 확인
SELECT FILE#, STATUS, NAME FROM V$DATAFILE;
-- STATUS가 ONLINE이어야 정상

-- 종료 (확인 후)
EXIT;
```

### 4-3. Tablespace 여유 공간 확인

```sql
-- sqlplus / as sysdba 접속 후

-- Tablespace 여유 공간 확인
SELECT
    ts.tablespace_name,
    ROUND(ts.bytes/1024/1024, 0) AS "전체(MB)",
    ROUND(NVL(fs.bytes,0)/1024/1024, 0) AS "여유(MB)",
    ROUND((1 - NVL(fs.bytes,0)/ts.bytes)*100, 1) AS "사용률(%)"
FROM
    (SELECT tablespace_name, SUM(bytes) bytes FROM dba_data_files GROUP BY tablespace_name) ts,
    (SELECT tablespace_name, SUM(bytes) bytes FROM dba_free_space GROUP BY tablespace_name) fs
WHERE ts.tablespace_name = fs.tablespace_name(+)
ORDER BY 4 DESC;
```

> ⚠️ **주의**: 사용률 **90% 이상**인 Tablespace는 작업 전 DBA에게 확인 요청.

### 4-4. Alert Log 확인

```bash
# Alert Log 위치 확인
sqlplus / as sysdba <<EOF
SELECT VALUE FROM V\$PARAMETER WHERE NAME = 'background_dump_dest';
EXIT;
EOF

# 또는 (Oracle 11g 이상)
# $ORACLE_BASE/diag/rdbms/<DB명>/<인스턴스명>/trace/alert_<인스턴스명>.log

# 최근 100줄 확인
tail -100 $ORACLE_BASE/diag/rdbms/orcl/orcl1/trace/alert_orcl1.log
# ※ 경로는 실제 환경에 맞게 수정

# ORA- 에러 확인
grep "ORA-" $ORACLE_BASE/diag/rdbms/orcl/orcl1/trace/alert_orcl1.log | tail -20
```

> ⚠️ **주의**: `ORA-00600`, `ORA-07445` 같은 Internal Error가 있으면 작업 전 DBA 확인 필수.

### 4-5. Redo Log 상태 확인

```sql
-- sqlplus / as sysdba 접속 후

-- Redo Log 상태 확인
SELECT GROUP#, STATUS, ARCHIVED, MEMBERS FROM V$LOG;
-- STATUS: CURRENT (현재 사용 중), ACTIVE (아카이브 대기), INACTIVE (비어있음)
-- ARCHIVED: YES 이어야 아카이브 정상

-- Redo Log 멤버 위치 확인
SELECT GROUP#, MEMBER FROM V$LOGFILE ORDER BY GROUP#;
```

## 5. JAVA WAS 점검

> WAS 종류(WebLogic, JBoss, Tomcat 등)에 따라 명령어가 다름.
> 아래는 공통적으로 확인 가능한 방법.

### 5-1. JAVA 프로세스 확인

```bash
# 현재 실행 중인 JAVA 프로세스 확인
ps -ef | grep java | grep -v grep

# 프로세스 수 확인
ps -ef | grep java | grep -v grep | wc -l

# JAVA 프로세스 PID 및 기동 경로 확인
ps -ef | grep java | grep -v grep | awk '{print $2, $8, $9}'
```

### 5-2. WAS 포트 확인

```bash
# WAS 리스닝 포트 확인 (일반적으로 8080, 8443, 7001 등)
netstat -an | grep LISTEN | grep -E "8080|8443|7001|7002"

# 접속 중인 세션 수
netstat -an | grep 8080 | grep ESTABLISHED | wc -l
```

> 🔴 **놓치기 쉬운 포인트**: 작업 전 WAS 접속 세션이 **0**이 되었는지 확인 후 중지.
> 세션이 있는 상태에서 강제 종료하면 처리 중이던 트랜잭션이 손실될 수 있음.

### 5-3. WAS 로그 확인

```bash
# WAS 로그 위치는 환경마다 다름. 아래는 일반적인 위치
# WebLogic: $DOMAIN_HOME/servers/<서버명>/logs/
# JBoss:    $JBOSS_HOME/server/<서버명>/log/
# Tomcat:   $CATALINA_HOME/logs/

# 최근 오류 확인
grep -i "ERROR\|Exception\|FATAL" <WAS_LOG_PATH>/server.log | tail -30
```

### 5-4. Heap 메모리 사용률 확인

```bash
# jstat으로 GC 상태 확인 (JAVA_PID는 위에서 확인한 PID)
# HP-UX에서 jstat 사용 가능 여부 확인 필요
which jstat && jstat -gc <JAVA_PID> 1000 5

# 간단하게 힙 정보만 확인
jstat -gcutil <JAVA_PID> 1000 5
# 출력: S0  S1   E    O    M   YGC YGCT  FGC FGCT  GCT
# O(Old Generation) 사용률이 80% 이상이면 주의
```

## 7. Cron 점검

> 🔴 **놓치기 쉬운 포인트 (매우 중요)**:
> 작업 시간대에 Cron이 실행되면 DB/WAS가 중지된 상태에서 배치가 돌아 **오류가 쌓이거나 데이터가 오염**될 수 있다.
> 반드시 사전에 파악하고, 작업 당일에는 Cron을 일시 중지해야 한다.

### 7-1. Cron 등록 현황 추출

```bash
# HP-UX 서버에서 수행
# 주요 계정의 crontab 확인 (root, oracle, app계정 등)

# root 계정 crontab
crontab -l

# 다른 계정 crontab 확인 (root 권한으로)
crontab -l -u oracle
crontab -l -u app    # 또는 실제 앱 계정명
crontab -l -u batch  # 배치 전용 계정이 있는 경우

# 전체 계정 crontab 일괄 확인 (root 권한)
for user in $(cut -f1 -d: /etc/passwd); do
    echo "=== $user ==="
    crontab -l -u $user 2>/dev/null
done

# 시스템 cron 디렉토리 확인
ls -la /etc/cron.d/ 2>/dev/null
ls -la /etc/cron.daily/ 2>/dev/null
ls -la /etc/cron.weekly/ 2>/dev/null
```

> 💡 **TIP**: 위 명령 결과를 파일로 저장해 둘 것.
> ```bash
> # 전체 crontab 백업
> for user in $(cut -f1 -d: /etc/passwd); do
>     crontab -l -u $user 2>/dev/null && echo "### $user ###"
> done > /tmp/crontab_backup_$(date +%Y%m%d).txt
> echo "백업 저장: /tmp/crontab_backup_$(date +%Y%m%d).txt"
> ```

### 7-2. 작업 시간대 배치 충돌 확인

```bash
# 예를 들어 작업이 오전 2시~4시라면
# 해당 시간대 Cron 확인
crontab -l | grep -E "^[0-9*]" | awk '{print $1, $2, $0}' | sort
# 1열: 분, 2열: 시간 → 작업 시간대(2~4시)와 겹치는 항목 파악
```

### 7-3. 작업 당일 Cron 일시 중지 방법

```bash
# 방법 1: crontab 전체 백업 후 비우기 (권장하지 않음 - 복원 실수 위험)
crontab -l > /tmp/crontab_bak_$(date +%Y%m%d).txt
crontab -r   # 위험: crontab 전체 삭제

# 방법 2 (권장): 각 항목을 주석 처리
# crontab -e 로 열어서 실행될 항목 앞에 # 추가
# 예: # 0 2 * * * /home/batch/run_daily.sh

# 방법 3: crond 데몬 자체를 일시 중지 (모든 cron 중지 - 주의 필요)
# HP-UX
/sbin/init.d/cron stop
# 복원
/sbin/init.d/cron start
```

> ⚠️ **주의**: 방법 3(crond 중지)은 해당 서버의 모든 Cron이 멈추므로
> 다른 중요 배치도 영향받는다. 사전에 영향 범위를 파악한 후 결정할 것.

## 8. 점검 결과 저장 방법

모든 점검 결과를 파일로 저장해 두어야 작업 후 비교할 수 있다.

```bash
# HP-UX Active 서버에서 한번에 저장
SAVE_DIR=/tmp/pre_check_$(date +%Y%m%d)
mkdir -p $SAVE_DIR

# OS 정보
uname -a > $SAVE_DIR/os_info.txt
date >> $SAVE_DIR/os_info.txt

# 디스크
bdf > $SAVE_DIR/disk_usage.txt

# 프로세스
ps -ef > $SAVE_DIR/process_list.txt

# 네트워크
netstat -an > $SAVE_DIR/network_status.txt

# ServiceGuard
cmviewcl -v > $SAVE_DIR/sg_status.txt 2>&1

# Cron 백업
for user in root oracle app batch; do
    echo "=== $user ===" >> $SAVE_DIR/crontab_all.txt
    crontab -l -u $user 2>/dev/null >> $SAVE_DIR/crontab_all.txt
done

echo "점검 결과 저장 완료: $SAVE_DIR"
ls -la $SAVE_DIR
```

> 💡 **TIP**: 저장된 파일을 로컬 PC 또는 AIX 백업 서버에 복사해 두면 더 안전하다.
> ```bash
> # AIX 백업 서버로 복사
> scp -r $SAVE_DIR oracle@<AIX_IP>:/backup/pre_check/
> ```
