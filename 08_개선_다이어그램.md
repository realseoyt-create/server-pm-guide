# 08. Failover 대응 방식 개선 — AS-IS vs TO-BE

> **핵심 변화**: 사후 발견(Reactive) → 사전 예방(Proactive)
>
> 비교 시점이 **Failover 이후**에서 **Failover 이전**으로 앞당겨진다.

---

## 무엇이 달라지는가

```mermaid
timeline
    title Failover 작업 타임라인 비교

        Failover  : 서비스 중지 / VIP 이동 / 서비스 기동
        작업 후    : ❌ 미기동 프로세스 발견
                   : ❌ 원인 조사 (시간 소요)

    section TO-BE (선제 대응)
        작업 전    : 프로세스 스냅샷
                   : Crontab 리스트 수집
                   : ✅ 교차 비교 → 고아 프로세스 식별
                   : ✅ 선제 조치 완료
        Failover  : 서비스 중지 / VIP 이동 / 서비스 기동
        작업 후    : ✅ 빠른 검증
                   : ✅ 보고
```

## 핵심 개념 — "고아 프로세스(Orphan Process)"

```mermaid
flowchart LR
    P["실행 중인 프로세스\n(ps -ef)"]
    C["Crontab 등록 목록\n(전 계정 수집)"]
    P --> CMP{교차 비교}
    C --> CMP

    CMP -->|"Crontab O\n= 관리되는 프로세스"| SAFE["✅ 안전\nFailover 후 Cron이\n자동 재기동"]
    CMP -->|"Crontab X\n= 고아 프로세스"| ORPHAN["⚠️ 고아 프로세스\nFailover 후 아무도\n재기동 안 함"]

    ORPHAN --> ACTION["선제 조치 필요\n① 담당자 확인\n② Crontab 등록\n③ 수동 기동 계획\n④ 불필요하면 종료"]

    style ORPHAN fill:#ffeeaa,stroke:#cc9900
    style SAFE fill:#ccffcc,stroke:#339933
    style ACTION fill:#fff0cc,stroke:#cc9900
```

## 개선 효과 비교

| 항목 | AS-IS | TO-BE |
|------|-------|-------|
| 비교 시점 | Failover **이후** | Failover **이전** |
| 문제 발견 시점 | 장애 발생 후 | 작업 전 사전 인지 |
| 대응 방식 | 사후 수동 조치 | 선제 계획 수립 |
| 작업 중 긴급 대응 | 빈번히 발생 | 최소화 |
| Failover 시간 예측 | 불확실 | 예측 가능 |
| 담당자 확인 시점 | Failover 후 (긴박한 상황) | Failover 전 (여유 있는 상황) |
| 미기동 프로세스 위험 | 높음 | 낮음 |

## 실행 방법

```bash
# STEP 1: Failover 전에 실행 (양산 영향 없음)
bash scripts/process_cron_compare.sh

# STEP 2: 출력된 고아 프로세스 목록 확인
cat /tmp/orphan_process_report_*.txt

# STEP 3: 각 고아 프로세스에 대해 담당자 확인 및 조치 결정
# → Crontab 등록 / 수동 기동 계획 / 불필요 시 종료

# STEP 4: 조치 완료 후 Failover 진행
```

> 스크립트 위치: `scripts/process_cron_compare.sh`
> 상세 설명: 스크립트 내 주석 참고
